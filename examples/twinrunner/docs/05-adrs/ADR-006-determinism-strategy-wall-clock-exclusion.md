# ADR-006 — Determinism Strategy: Wall-Clock Exclusion from Checksums and Fixed/Seeded Time in Tests

> **Stage 5 — Architecture Decision Record** (spec §15.5). Streams; only genuinely irreversible
> decisions reach the human (§8). One file per decision; lives in `docs/05-adrs/ADR-NNN-*.md`.
> Each ADR must link to the REQ-IDs and components it serves. Non-technical users receive the
> decision framed as a plain tradeoff.

**Decision summary:** `BuildArtifact` checksums and `FlashJob` verify-after-write results are
computed exclusively from operation inputs (job parameters, simulator steps, output content) —
wall-clock time, UUIDs, and timestamps are explicitly excluded from checksum inputs; simulator
progress steps are a fixed deterministic sequence driven by job inputs, not by elapsed time;
timestamps for `LogEntry` records are fixed/seeded in tests — because REQ-NFR-005 ("identical
inputs → identical progress sequences, results, and checksums") cannot be satisfied if any
time-sensitive value enters the reproducible outputs.

## Context

TwinRunner's top success measure (from `01-requirements.md`) is:

> "Given a known-good 64 MB NAND dump fixture, TwinRunner parses it and extracts CPU key +
> console type + bootloader versions, and a simulated flash of a 64 MB image reports deterministic
> 0→100% progress with a verifiable result."

REQ-NFR-005 states: "identical inputs always yield identical progress sequences, results,
checksums, and logs — so behavior is reproducible for tests and demos without live hardware or
network."

REQ-019 and REQ-023 require that build/flash operations complete with a **verifiable result**
including a **deterministic checksum** (build) and a **verify-after-write confirmation** (flash).

The risk, noted explicitly in `04-architecture.md` as ARCH-RISK-005: "Wall-clock time or any
time-based value must not leak into `BuildArtifact` checksums or progress sequences, or
RULE-007/008 break." Domain Rules RULE-007 and RULE-008 state that simulator outputs are
deterministic given the same inputs.

Specific failure modes if this decision is not made explicitly:

1. **Checksum non-determinism.** If `BuildArtifact.checksum` is computed from content that
   includes the current timestamp (e.g., a header field like "built at 2026-06-10T14:32:01Z"),
   two identical build runs produce different checksums and the "verifiable result" assertion
   fails.

2. **Progress sequence non-determinism.** If the simulator's stepped progress is driven by
   `std::time::Instant` (advancing by real elapsed time), the progress sequence is
   hardware-speed-dependent and non-reproducible in tests.

3. **`LogEntry` timestamp leakage.** If test assertions compare serialized log entries
   including ISO-8601 timestamps produced by `SystemTime::now()`, the assertions fail on every
   run unless the time is seeded or stripped from comparisons.

4. **`KeyRecord`/`BuildJob`/`FlashJob` UUID leakage.** If determinism tests compare record IDs
   generated by `uuid::Uuid::new_v4()` (random), they fail every run. Record IDs must be
   excluded from any comparison that asserts determinism of functional outputs.

This decision is irreversible at architecture level because the checksum input set is baked into
the contract of `BuildArtifact` and the `FlashJob` verify result. Changing which fields enter the
checksum after slices are built requires updating every test that asserts a specific checksum
value and every downstream consumer that stores/compares artifact checksums.

**Relevant REQ-IDs:** REQ-019, REQ-023, REQ-NFR-005, REQ-NFR-006; RULE-007, RULE-008
**Components affected:** `twinrunner-core::build`, `twinrunner-core::flash`,
`twinrunner-core::log`, `twinrunner-core::keys`, `twinrunner::worker`

## Decision

The determinism strategy is composed of four interlocking rules, all enforced in
`twinrunner-core`:

**Rule D-1 — Checksum inputs are time-free.**
`BuildArtifact.checksum` is computed by hashing:
- the source dump's content bytes (read via `twinrunner-core::nand`),
- the selected `TimingFile` identifier (a deterministic string, not its filesystem mtime),
- the artifact type (ECC or XeLL),
- any other build-parameter fields that affect the output image structure.
Wall-clock timestamps, UUIDs, and filesystem metadata are **not** checksum inputs.

**Rule D-2 — Simulator progress is step-count driven, not time driven.**
The `SimulatorBuildBackend` and `SimulatorFlashBackend` advance through a fixed, deterministic
sequence of steps (e.g., 10 steps → 0%, 10%, 20%, …, 100%) driven by the job's input parameters
(image size class, operation type). The step count and step size are derived from inputs, not from
elapsed wall-clock time. `std::time::Instant` is never consulted inside the simulator.

**Rule D-3 — `LogEntry` timestamps are injected, not read from the system clock in tests.**
`twinrunner-core::log` accepts timestamps as parameters (or via an injectable clock abstraction)
rather than calling `SystemTime::now()` directly in production code. In determinism tests, a
fixed seed timestamp (e.g., `2000-01-01T00:00:00Z`) is injected. In production, `SystemTime::now()`
is called once at the `LogEntry` construction site in the binary shell or via the clock abstraction.
This ensures test assertions on `LogEntry` content are time-stable.

**Rule D-4 — Record IDs (UUIDs) are excluded from determinism-relevant comparisons.**
`KeyRecord`, `BuildJob`, and `FlashJob` use `uuid::Uuid::new_v4()` for identity. These IDs are
excluded from any assertion that tests determinism of functional outputs (checksums, progress
sequences, extracted console info). Tests that need stable IDs use injected fixed UUIDs; tests
that assert functional correctness avoid comparing IDs.

> **Chosen:** wall-clock and random values excluded from checksum inputs; step-count-driven
> simulator progress; injectable clock for log timestamps; UUID injection in determinism tests

*Human gate triggered:* no — this decision is architecture-derived from the determinism
requirements (REQ-NFR-005, RULE-007/008) and the success criteria in `01-requirements.md`. It is
significant and the exclusion-set is costly to change once baked into test corpora, but it is not
a product-meaningful user choice. Recorded as Accepted per the approved architecture
(ARCH-RISK-005).

## Consequences

### Positive

- The "verifiable result" promise in REQ-019 and REQ-023 is concretely fulfillable: a test can
  assert `artifact.checksum == EXPECTED_CHECKSUM` and the assertion passes on every run, on
  every machine, regardless of wall-clock time (REQ-NFR-005, REQ-NFR-006).
- RULE-007 ("simulator build output is deterministic given the same inputs") and RULE-008
  ("simulator flash output is deterministic given the same inputs") are mechanically testable:
  run the simulator twice with the same `BuildJob`/`FlashOperation` and assert the checksums
  are equal.
- The progress sequence is predictable: given a 64 MB flash write operation, the simulator
  always emits exactly N `FlashProgressed { pct }` events in a fixed order. Tests that assert
  on progress event sequences are stable.
- Injectable timestamps (Rule D-3) mean `LogEntry` serialization tests are stable without
  mocking `SystemTime` globally, avoiding the fragile global-clock-mock pattern.
- Deterministic behavior makes demos reproducible: a live demo of TwinRunner always shows the
  same step sequence, the same checksum, and the same result for the same input fixture — there
  is no randomness that could make a demo go differently than rehearsed.

### Negative

- **The injectable clock abstraction adds boilerplate.** `twinrunner-core::log`'s `LogEntry`
  construction must accept a timestamp parameter or a clock trait rather than calling
  `SystemTime::now()` directly. This is a small amount of indirection that adds a seam to every
  log entry creation site.
- **Checksum exclusion rules must be documented and enforced by convention.** There is no Rust
  type-system mechanism that prevents a future developer from adding a timestamp field to
  `BuildArtifact` and including it in the checksum computation. A comment in the code and a
  note in `06-technical-design.md` are the enforcement mechanisms; a CI lint rule could
  strengthen this but adds tooling cost.
- **The step-count-driven progress model does not model real timing fidelity.** The simulator's
  deterministic steps do not reflect real flash/build operation timing (a 512 MB flash takes
  longer than a 16 MB flash in real life, but the number of progress events may be the same in
  the simulator). Users who are familiar with real programmer hardware may find the progress
  cadence unrealistic. This is acceptable for an example but would be a limitation if real
  timing fidelity were required.
- **Fixed/seeded UUIDs in determinism tests increase test-harness complexity.** Tests that need
  stable record IDs must either inject a factory that produces fixed UUIDs or avoid asserting on
  IDs entirely. Test harnesses must document which IDs are fixed and which are random per run.

### Future obligations

- `docs/06-technical-design.md` must document the checksum input specification for `BuildArtifact`
  (which fields enter the hash, in what order, with what encoding) so the `twinrunner-core::build`
  implementation is unambiguous.
- `docs/06-technical-design.md` must document the step-count derivation formula for
  `SimulatorBuildBackend` and `SimulatorFlashBackend` (e.g., steps = f(SizeClass, operation_type)).
- `docs/07-contracts.md` must specify the `BuildArtifact` schema including the checksum field and
  note its determinism contract.
- `docs/08-test-strategy.md` must include a test asserting `checksum(run1) == checksum(run2)` for
  the same inputs, exercising RULE-007 and RULE-008.
- Any future addition to `BuildArtifact` or `FlashJob` result types must be evaluated against
  Rule D-1 before including the field in the checksum input.

## Alternatives Considered

### Option A — Exclude wall-clock/time from checksum inputs; step-count-driven simulator; injectable clock *(chosen)*

Determinism enforced by design. Chosen — see Decision above.

### Option B — Include timestamps in `BuildArtifact` metadata but exclude from checksum hash

- **What it is:** `BuildArtifact` carries a `built_at: DateTime<Utc>` field in its metadata, and
  the checksum is computed from the content bytes only (excluding the metadata). Tests assert on
  the checksum value; they do not compare the `built_at` field.
- **Why this is partially acceptable but incomplete:** this approach handles Rule D-1 (checksum
  excludes timestamps) but does not address Rule D-2 (step-count driven progress) or Rule D-3
  (log timestamp injection). If the progress steps are time-driven, tests that assert on the
  sequence of `BuildProgressed` events are still non-deterministic. Additionally, having a
  timestamp in `BuildArtifact` metadata that is not covered by the checksum creates a consistency
  gap: two artifacts with the same checksum can have different `built_at` values, meaning the
  "verifiable result" is not a complete proof that the artifact matches. This option is a
  partial solution; the chosen approach (exclude timestamps from all checksum-relevant paths) is
  more complete.
- **Would be right if:** a timestamp in the artifact metadata were required by a downstream
  consumer (e.g., a real build system that logs artifact provenance). No such consumer exists in
  the current scope.

### Option C — Non-deterministic simulator (real-time progress, random checksums per run)

- **What it is:** the simulator drives progress using `std::time::Instant` (e.g., emit a progress
  event every 50 ms); `BuildArtifact.checksum` includes the current timestamp. Each run produces
  a different checksum.
- **Why rejected:** this directly violates REQ-NFR-005 ("identical inputs always yield identical
  checksums") and makes the top success measure ("verifiable result") untestable — there is no
  EXPECTED_CHECKSUM to assert against. The project's acceptance criteria cannot be met. This is
  not a genuine tradeoff; it fails a hard requirement and is listed only to make explicit that
  real-time-driven progress is not an option here.
- **Would be right if:** the requirement were "show realistic timing variability" rather than
  "deterministic and reproducible." No such requirement exists in TwinRunner's scope.

### Option D — Mock `SystemTime::now()` globally in tests

- **What it is:** use a global-state clock mock (via a `thread_local` or a static `AtomicU64`
  that tests override) so that all calls to `SystemTime::now()` in production code return a
  fixed value in test contexts.
- **Why rejected:** global-state mocks create test ordering dependencies (one test's clock state
  affects another's) and are fragile in parallel test execution (Rust's default `cargo test` runs
  tests in parallel threads). The injectable clock abstraction (Rule D-3) is a local, dependency-
  injected approach that is immune to parallel-execution issues and does not require modifying
  global state. Global mocking is a last resort when the production code cannot be refactored;
  since `twinrunner-core` is being designed from scratch, the injectable approach is strictly
  better.
- **Would be right if:** `twinrunner-core` were a third-party library whose source could not be
  modified, requiring external mocking. Since the library is first-party and designed here,
  injection is the correct approach.
